1. 请详细解释 HTTP 1.0 1.1、HTTP 2、HTTP 3 的主要区别，以及它们在实际项目中对性能的影响。
答：HTTP 1.0 是最早的 HTTP 版本，它的每个请求都需要建立一个新的 TCP 连接，请求完成后连接就会关闭。这会导致多个请求的延迟和资源浪费。
HTTP 1.1 引入了持久连接（keep-alive）和管道请求（pipelining），可以在一个 TCP 连接上发送多个请求，减少了连接的建立和关闭次数，提高了性能。但是http1.1建立连接后，每个请求都需要等待前一个请求的响应，这会导致请求的延迟。
HTTP 2 是 HTTP 1.1 的升级版本，它基于二进制分帧层和流的概念，实现了多路复用、头部压缩、服务器推送等功能，进一步提高了性能。
HTTP 3 是 HTTP 2 的升级版本，它基于 QUIC 协议，实现了低延迟、高吞吐量的传输，同时支持零延迟的连接建立和恢复。
在实际项目中，HTTP 2 和 HTTP 3 都有较大的性能优势，而 HTTP 1.1 则相对较慢。
2. 什么是闭包？闭包的形成条件是什么？请举例说明闭包在实际开发中的应用场景，以及使用闭包可能带来的问题。
答：闭包是指在一个函数内部定义的函数，它可以访问外部函数的变量和参数，即使外部函数已经执行完毕。
闭包的形成条件是：内部函数必须引用外部函数的变量或参数，并且外部函数必须返回内部函数。
闭包在实际开发中的应用场景有：
模块模式：将相关的变量和函数封装在一个闭包中，实现了封装和隐藏，避免了全局变量的污染。
回调函数：将一个函数作为参数传递给另一个函数，在异步操作完成后调用。
闭包可以用于创建私有变量和方法，实现了封装和隐藏，避免了全局变量的污染。
使用闭包可能带来的问题有：
内存泄漏：如果闭包引用了外部函数的变量，而外部函数的变量又被闭包引用，那么这两个变量就会一直存在于内存中，直到闭包被销毁。

3. 请对比 var、let、const 的区别，包括变量提升、作用域、重复声明、修改值等方面，并用代码示例验证。
答：var、let、const 都是用来声明变量的关键字，但是它们之间有一些区别。
变量提升：var 声明的变量会被提升到函数顶部，而 let、const 声明的变量不会被提升。
作用域：var 声明的变量是函数作用域的，而 let、const 声明的变量是块级作用域的。
重复声明：var 可以重复声明同一个变量，而 let、const 不可以重复声明同一个变量。
修改值：var 声明的变量的值可以被修改，而 let、const 声明的变量的值不能被修改。
代码示例：
var a = 1;
let b = 2;
const c = 3;
a = 4; // 可以修改
b = 5; // 可以修改
c = 6; // 报错，const 声明的变量的值不能被修改

4. 什么是浏览器的重排（Reflow）和重绘（Repaint）？列举至少 5 种会触发重排的操作，以及在项目中如何减少重排和重绘以优化性能。
答：浏览器的重排（Reflow）和重绘（Repaint）是指浏览器在渲染页面时，根据元素的位置、大小、样式等信息，重新计算元素的布局和绘制元素的过程。
重排（Reflow）：当元素的位置或大小发生变化时，浏览器需要重新计算元素的布局，这个过程称为重排。
重绘（Repaint）：当元素的样式发生变化时，浏览器需要重新绘制元素，这个过程称为重绘。
会触发重排的操作有：
修改元素的位置、大小、样式等属性
添加或删除 DOM 元素
修改元素的内容
浏览器窗口大小变化
在项目中如何减少重排和重绘以优化性能？
答：减少重排和重绘的方法有：

减少重排和重绘的次数：
避免频繁操作样式，尽量一次性将样式写好并应用到元素上。
避免频繁操作DOM，尤其是添加、删除和修改大量元素。可以使用DocumentFragment或虚拟DOM等技术来优化。
避免使用table布局，因为table元素在改变时会触发整个table的重排。

利用CSS3动画代替JavaScript动画：
CSS3动画可以由浏览器的GPU来优化处理，减少CPU的负担。
使用transform、opacity等属性进行动画，它们不会触发重排，只会触发重绘。

使用请求动画帧（requestAnimationFrame）：
requestAnimationFrame可以将动画的执行时间调整到浏览器的下一次重绘之前，从而实现平滑动画效果。
它可以避免不必要的重排和重绘，提高动画性能。

避免使用内联样式：
内联样式会导致浏览器每次都需要解析和应用样式，增加重排和重绘的次数。应该尽量使用外部CSS文件来定义样式。

优化CSS选择器：
复杂的CSS选择器会增加浏览器的计算负担，应该尽量简化选择器，提高选择器的效率。

使用CSS的will-change属性：
will-change属性可以告诉浏览器某个元素即将发生变化，让浏览器提前做好准备，从而减少重排和重绘的开销。但需要注意，过度使用will-change属性可能会导致额外的性能开销，因此应该谨慎使用。

避免不必要的布局计算：
使用visibility: hidden而不是display: none来隐藏元素，因为前者只会触发重绘而后者会触发重排。
避免使用offsetLeft、offsetTop等属性，因为它们会触发重排。如果需要获取这些值，可以先将其赋值给一个变量，然后再进行操作。

2. 阐述浏览器的事件流机制（捕获阶段、目标阶段、冒泡阶段），并实现一个案例：给父元素和子元素都绑定点击事件，通过代码控制只触发子元素的事件，不触发父元素的事件。

答：
浏览器的事件流机制包括捕获阶段、目标阶段和冒泡阶段。
捕获阶段：事件从window对象开始，向下传递到目标元素的祖先元素，依次触发祖先元素的事件处理函数。
目标阶段：事件到达目标元素，触发目标元素的事件处理函数。
冒泡阶段：事件从目标元素开始，向上传递到window对象，依次触发祖先元素的事件处理函数。
实现案例：
const parent = document.querySelector('.parent')
const child = document.querySelector('.child')
parent.addEventListener('click', (e) => {
    console.log('parent click')
})
child.addEventListener('click', (e) => {
    e.stopPropagation()
    console.log('child click')
})

答：
虚拟 DOM 原理：
虚拟 DOM 是 React 中用于描述真实 DOM 结构的 JavaScript 对象。当组件的状态或属性发生变化时，React 会根据新的状态或属性创建一个新的虚拟 DOM 树。
Diff 算法核心逻辑：
Key 的作用：Key 是 React 中用于标识列表中元素的唯一标识符。当列表中的元素发生变化时，React 会根据 Key 来判断哪些元素是新增、删除还是移动的，从而更高效地更新 DOM。
同层比较策略：React 会对虚拟 DOM 树进行同层比较，即只比较同一层级的元素，而不是递归比较整个树。这样可以减少比较的次数，提高比较效率。
虚拟 DOM 如何提升页面渲染性能：
虚拟 DOM 可以避免频繁的真实 DOM 操作，减少页面的重绘和重排次数，从而提升页面的渲染性能。

2. 对比 React 中的类组件和函数组件，包括状态管理、生命周期、代码复用方式（如 HOC、Render Props、Hook）等方面，说明为什么现在更推荐使用函数组件。
答：
状态管理：
类组件：使用 this.state 来管理状态，通过 setState 方法来更新状态。
函数组件：使用 useState Hook 来管理状态，通过调用 useState 函数返回的 setState 函数来更新状态。
生命周期：
类组件：使用生命周期方法（如 componentDidMount、componentDidUpdate、componentWillUnmount 等）来处理组件的生命周期事件。
函数组件：使用 useEffect Hook 来处理组件的生命周期事件，通过传递不同的依赖数组来控制 useEffect 函数的执行时机。
代码复用方式：
类组件：使用 HOC（Higher-Order Component）、Render Props 等方式来实现代码复用。
函数组件：使用 Hook（如 useState、useEffect 等）来实现代码复用。
为什么现在更推荐使用函数组件：
函数组件更简洁、更易于理解和维护。

useState：
useState 用于在函数组件中添加状态。它接受一个初始状态值作为参数，并返回一个数组，数组的第一个元素是当前状态值，第二个元素是更新状态的函数。
useEffect：

useMemo：

useCallback：
useCallback 用于在函数组件中缓存回调函数，避免重复创建。它接受一个回调函数和一个依赖数组作为参数。当依赖数组中的值发生变化时，回调函数会被重新创建。

5. 阐述 React 状态管理方案（如 Context + useReducer、Redux、MobX、Zustand 等），对比 Redux 和 Zustand 的优缺点，并用 Zustand 实现一个简单的计数器状态管理案例。
答：
React 状态管理方案：
Context + useReducer：
Context 提供了一种在组件树中共享数据的方式，而 useReducer 则用于管理状态的更新。
Redux：
Redux 是一个基于 Flux 架构的状态管理库，它使用一个全局的状态树来管理应用的状态。
MobX：
MobX 是一个基于响应式编程的状态管理库，它使用装饰器或 Observable 来定义可观察的状态，当状态发生变化时，相关的组件会自动重新渲染。
Zustand：
Zustand 是一个基于 React Hooks 的状态管理库，它使用简单的 API 来定义状态和更新状态的函数。
对比 Redux 和 Zustand：
Redux：
优点：
集中式状态管理，方便调试和维护。
支持中间件和异步操作。
缺点：
学习曲线陡峭，需要理解 Flux 架构和 Redux 原理。
配置复杂，需要编写 reducer、action、middleware 等代码。
Zustand：
优点：
简单易用，API 设计简单，学习曲线平缓。
无需配置，无需编写 reducer、action 等代码。
支持异步操作，无需使用 middleware。
缺点：
不支持中间件，无法实现一些复杂的功能。
不支持时间旅行调试，无法回溯到之前的状态。

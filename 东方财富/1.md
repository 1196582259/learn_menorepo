一、技术基础模块（一面必问，占比 30%）

1. XHR、Ajax、Axios、Fetch 的区别
   考察方向：网络请求底层原理与库特性
   答题思路：
   概念关联：Ajax 是异步请求技术统称，XHR 是 Ajax 的核心实现方式（原生 API）；
   库的优势：Axios 基于 XHR 封装，支持拦截器、取消请求、请求 / 响应转换，适配金融场景的异常处理；
   Fetch 区别：原生 API，基于 Promise，无拦截器、取消请求需手动实现，错误处理仅捕获网络错误（4xx/5xx 不触发 reject）；
   金融场景选型：优先 Axios，需处理行情请求的取消、超时重试等需求。
2. HTTP 1.0/1.1/2.0/3.0 的核心差异
   考察方向：协议演进与性能优化
   答题思路：
   HTTP 1.0：无长连接，每次请求需建立 TCP 连接，效率低；
   HTTP 1.1：支持长连接（Connection: keep-alive）、管线化请求、缓存控制（Cache-Control），解决 1.0 的连接开销问题；
   HTTP 2.0：二进制帧、多路复用（同一连接并发请求）、头部压缩、服务器推送，大幅提升传输效率，适配行情高频请求；
   HTTP 3.0：基于 QUIC 协议（UDP 封装），解决 TCP 队头阻塞，降低连接建立延迟，适合金融数据实时传输。
3. TCP 三次握手、四次挥手流程，UDP 与 TCP 的差异
   考察方向：传输层协议原理
   答题思路：
   三次握手：客户端发送 SYN→服务端回复 SYN+ACK→客户端发送 ACK，目的是建立可靠连接，同步序列号；
   四次挥手：客户端发送 FIN→服务端回复 ACK→服务端发送 FIN→客户端回复 ACK，因服务端可能仍有数据需传输，需分两步确认关闭；
   差异对比：TCP 可靠、面向连接、有序、有流量控制，适合金融交易数据传输；UDP 不可靠、无连接、无序、效率高，适合行情实时推送（允许少量丢包）。
4. var/let/const 的区别，变量提升与 this 指向
   考察方向：JS 基础语法与作用域
   答题思路：
   声明特性：var 存在变量提升、函数级作用域、可重复声明；let/const 块级作用域、无变量提升、不可重复声明，const 声明时需赋值且不可修改引用；
   变量提升：var 提升后初始值为 undefined，let/const 存在暂时性死区（TDZ），未声明前使用报错；
   this 指向：全局环境 this 指向 window（非严格模式）/undefined（严格模式）；函数内 this 取决于调用方式（普通调用指向 window、对象调用指向对象、bind/call/apply 手动绑定）。
5. 浏览器渲染流程，JS 是否阻塞渲染？为什么？
   考察方向：浏览器底层原理
   答题思路：
   渲染流程：解析 HTML 生成 DOM 树→解析 CSS 生成 CSSOM 树→合并为渲染树→布局（Layout）→绘制（Paint）→合成（Composite）；
   阻塞逻辑：JS 会阻塞渲染，因 JS 可操作 DOM 和 CSSOM（如 document.write 修改 DOM），浏览器需等待 JS 执行完成后再继续构建渲染树；
   优化方案：script 标签加 defer（延迟执行，不阻塞 HTML 解析）或 async（异步执行，加载完成后立即执行），金融行情页面需优先加载核心 JS。
6. 垂直居中的 5 种实现方式
   考察方向：CSS 布局实战
   答题思路：
   Flex 布局：父容器设置 display: flex; align-items: center; justify-content: center;（最常用）；
   Grid 布局：父容器 display: grid; place-items: center;；
   定位 + transform：子元素 position: absolute; top: 50%; left: 50%; transform: translate (-50%, -50%);；
   table-cell 布局：父容器 display: table-cell; vertical-align: middle; text-align: center;；
   已知高度：子元素 position: absolute; top: 0; bottom: 0; margin: auto;。

7. Vue 响应式原理（Object.defineProperty/Proxy 的差异）
   考察方向：框架核心机制
   答题思路：
   Vue 2.x 实现：基于 Object.defineProperty，遍历 data 属性设置 getter/setter，收集依赖（Watcher），数据变化时触发更新；
   Vue 3.x 实现：基于 Proxy，直接代理整个 data 对象，支持动态添加属性、数组索引修改，无需像 Vue 2.x 那样手动 $set；
   差异对比：Proxy 支持更多数据类型（Map/Set）、性能更优、无需递归遍历子属性，但存在浏览器兼容性问题（不支持 IE）。
8. Vue 路由 hash/history 模式的底层实现
   考察方向：前端路由原理
   答题思路：
   hash 模式：基于 URL 中的 #（锚点），通过 window.onhashchange 监听 URL 变化，无后端配置依赖，缺点是 URL 含 #不够美观；
   history 模式：基于 HTML5 History API（pushState/replaceState），可修改 URL 且不刷新页面，需后端配置所有路由指向 index.html（否则刷新 404），金融项目常用此模式优化 URL 体验。
9. React 中 useRef 与 useState 的区别，K 线图中如何选型？
   考察方向：Hook 实战与场景适配
   答题思路：
   核心区别：useState 赋值触发组件重渲染，useRef 修改 current 属性不触发；useRef 存储跨渲染周期的持久化数据，不参与组件 diff；
   K 线图选型场景：用 useRef 存储 ECharts/Canvas 实例（避免每次渲染重新初始化）、鼠标拖拽坐标、行情轮询定时器，减少不必要的重渲染。

12. Webpack 打包流程、Tree-shaking 原理
    考察方向：构建工具深度理解

function isSameWeek(date1Str, date2Str) {
// 解析日期字符串为Date对象
const parseDate = (str) => {
const year = Number(str.slice(0, 4));
const month = Number(str.slice(4, 6)) - 1; // 月份从0开始
const day = Number(str.slice(6, 8));
return new Date(year, month, day);
};

const date1 = parseDate(date1Str);
const date2 = parseDate(date2Str);

// 获取日期对应的星期（0=周日，1=周一...6=周六），转为以周一为第一天
const getWeekDay = (date) => {
let day = date.getDay();
return day === 0 ? 6 : day - 1;
};

// 计算两个日期的时间差（天）
const diffDays = Math.abs(Math.floor((date2 - date1) / (24 _ 60 _ 60 \* 1000)));
// 计算两个日期在本周的偏移量
const weekDay1 = getWeekDay(date1);
const weekDay2 = getWeekDay(date2);

// 同一周的条件：时间差 ≤ 两者偏移量的最大值，且不跨周
return diffDays <= Math.max(weekDay1, weekDay2) && Math.abs(weekDay1 - weekDay2) <= diffDays;

function climbStairs(n, memo = {}) {
if (n <= 2) return n;
if (memo[n]) return memo[n];
memo[n] = climbStairs(n - 1, memo) + climbStairs(n - 2, memo);
return memo[n];
}

// 迭代版（空间复杂度O(1)）
function climbStairsIter(n) {
if (n <= 2) return n;

for (let i = 3; i <= n; i++) {
[a, b] = [b, a + b];
}
return b;

1. 金融大数计算精度问题如何解决？
   考察方向：精度处理与库选型
   答题思路：
   问题根源：JS 浮点数采用 64 位双精度存储，无法精确表示 0.1、0.0001 等小数，导致金融计算（如股票价格 0.0001 精度加减）出错；
   解决方案：优先使用 decimal.js/big.js 库，通过字符串传入数值避免精度丢失，示例：new Decimal ('0.1').plus ('0.2').toFixed (4)；
   兜底方案：将小数转为整数计算（如 0.0001→1，计算后除以 10000 还原），禁止使用 toFixed（四舍五入规则不统一）。
2. WebSocket 在行情推送中的心跳重连机制
   考察方向：实时数据稳定性
   答题思路：
   心跳机制：客户端定期发送心跳包（如每 30 秒发送 {type: 'ping'}），服务端回复 pong，确认连接正常；
   重连逻辑：设置超时时间（如 10 秒），未收到 pong 响应则触发重连，重连时采用指数退避策略（1s、3s、5s 间隔），避免频繁重试；
   异常处理：断网时监听 navigator.onLine 事件，网络恢复后自动重连，重连成功后重新订阅行情数据。

4. 接口请求签名机制（防止伪造）
   考察方向：金融数据安全
   答题思路：
   签名规则：客户端生成 timestamp（时间戳）、nonce（随机字符串），结合 token 和请求参数，按固定顺序拼接后用 MD5/SHA256 加密生成签名；
   服务端校验：验证 timestamp 是否在有效时间内（如 5 分钟），防止过期请求复用；验证 nonce 唯一性，防止重复请求；重新计算签名与客户端传入的对比，不一致则拒绝请求；
   金融场景适配：交易接口必须加签名，行情接口可选，签名密钥需定期更新，避免泄露。

5. Node.js 是否多线程？如何解决并发？
   考察方向：后端能力与并发模型
   答题思路：
   核心机制：Node.js 主线程是单线程，基于事件循环处理异步任务（I/O 密集型高效，CPU 密集型受限）；
   并发解决方案：CPU 密集型任务用 worker_threads（创建工作线程，避免阻塞主线程）；多进程用 cluster 模块（ fork 多个子进程，利用多核 CPU）；
   金融场景应用：用 Node.js 搭建中间层，处理行情数据转发、接口签名校验，通过 cluster 模块提高并发处理能力。
6. Express 中间件的运行流程与错误处理
   考察方向：后端框架使用
   答题思路：
   运行流程：中间件是函数数组，按注册顺序执行，每个中间件接收 req、res、next 参数，调用 next () 进入下一个中间件，未调用 next () 则终止流程；
   错误中间件：需定义 4 个参数（err、req、res、next），注册在所有中间件之后，捕获前面中间件抛出的错误，返回统一错误格式（如 {code: 500, msg: ' 服务器错误 '}）；

7. 小程序与 H5 渲染的区别
   考察方向：跨端开发认知
   答题思路：
   渲染方式：H5 基于浏览器渲染（WebView），小程序采用双线程渲染（逻辑线程 + 视图线程），视图线程由原生渲染，性能更优；

   金融场景适配：交易功能优先用小程序（原生渲染更稳定、支持微信支付集成），行情展示可用 H5（适配多平台）。
